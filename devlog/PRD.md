# Product Requirements Document (PRD)

## Project: Private Text and File Sharing App

---

## 1. Overview

A lightweight web application that lets a user share text or files with another person using an encrypted link.

The sender creates a secret, the app encrypts it before storage, and the recipient can decrypt it using either:

- a password shared separately, or
- a generated decryption key shared separately

The product goal is simple private sharing, not chat, collaboration, or long-term cloud storage.

---

## 2. Product Goal

Build a full-stack app where:

- a sender can write text or upload a file
- the app stores only encrypted content
- the sender gets a shareable link
- the recipient opens the link and enters the password or decryption key
- the recipient sees the original text or downloads the original file

This should feel fast and simple, with no account system in v1.

---

## 3. Core Concept

The app is for one-way private sharing.

Typical flow:

1. Sender writes text or uploads a file
2. Sender chooses expiration and optional one-time view
3. App encrypts the content
4. App stores the encrypted payload
5. App returns a shareable link
6. Sender gives the recipient:
   - the link
   - the password or generated decryption key
7. Recipient opens the link and decrypts the content

The important product rule is:

**The server should not store or expose readable plaintext content.**

## 7. MVP Scope

### 7.1 Create Share

Sender can create a share with:

- Text content, or
- One uploaded file

Sender can configure:

- Expiration time
- Burn-after-read / one-time access
- Protection method:
  - password provided by sender, or
  - generated decryption key

### 7.2 Retrieve Share

Recipient can:

- open the link
- enter the password or decryption key
- decrypt the content
- read the text or download the file

### 7.3 Expiration and Deletion

The backend must make the share unavailable when:

- it has expired, or
- it has been consumed and burn-after-read is enabled

---

## 8. User Flows

### Create Text Share

1. User enters text
2. User selects expiration
3. User optionally enables burn-after-read
4. User chooses password or generated key
5. App encrypts the text
6. App uploads encrypted payload to backend
7. Backend returns share ID
8. UI shows the share link and instructions to send password/key separately

### Create File Share

1. User selects a file
2. User selects expiration
3. User optionally enables burn-after-read
4. User chooses password or generated key
5. App encrypts the file
6. App uploads encrypted payload plus file metadata
7. Backend returns share ID
8. UI shows the share link and instructions to send password/key separately

### Open Share

1. Recipient opens the link
2. App requests the password or decryption key
3. App fetches encrypted payload from backend
4. App decrypts the payload
5. App renders text or prepares the file download

### Error States

- Link not found
- Link expired
- Link already used
- Wrong password or invalid key
- Corrupted payload
- File too large

---

## 9. Functional Requirements

### 9.1 Content Types

The app must support:

- plain text shares
- single file shares

Future versions may support:

- multiple files
- folders
- rich text notes

### 9.2 Share Options

Each share must support:

- expiration time
- burn-after-read
- password or generated key

### 9.3 Delivery

The app must generate:

- a unique share link
- clear UI telling the sender to share the password/key separately

### 9.4 Viewing

The recipient must be able to:

- decrypt text and view it in the browser
- decrypt a file and download it

---

## 10. Product Rules

- The app must never display plaintext from backend storage directly
- The backend stores encrypted payloads and metadata only
- The decryption secret must be provided by the sender or generated by the app
- The sender is responsible for sharing the link and the password/key through separate channels if desired

---

## 11. Technical Direction

### Frontend

- React
- Vite
- Browser-based encryption/decryption

Responsibilities:

- collect text/file input
- encrypt before upload
- manage password or generated key flow
- decrypt on recipient side
- provide sender and recipient UX

### Backend

- FastAPI

Responsibilities:

- store encrypted payloads
- return encrypted payloads
- manage expiration and one-time access
- enforce payload size and metadata validation

### Database

- PostgreSQL

Stores:

- share metadata
- encrypted text payload or file reference
- expiration and deletion state

### File Storage

For files, choose one of:

- local storage for development
- S3-compatible object storage for production

Stored file objects must remain encrypted.

---

## 12. Suggested Data Model

### Share

- `id`
- `type` (`text` or `file`)
- `encrypted_payload`
- `file_name` (nullable)
- `file_size` (nullable)
- `mime_type` (nullable)
- `created_at`
- `expires_at`
- `burn_after_read`
- `is_deleted`
- `read_at` (nullable)

If file uploads are stored outside the database, also include:

- `storage_path` or `object_key`

---

## 13. API Requirements

### POST /shares

Creates a new encrypted share.

Request should include:

- encrypted payload
- content type
- file metadata if applicable
- expiration selection
- burn-after-read flag

Response should include:

- share ID

### GET /shares/{id}

Returns encrypted share payload and metadata if still valid.

Response should include:

- encrypted payload
- content type
- file metadata if applicable
- burn-after-read status

### Optional Future APIs

- `DELETE /shares/{id}`
- admin or cleanup endpoints

## 15. Security Expectations

This is an encrypted sharing app, so privacy is a core feature, but the PRD should stay product-focused.

Practical expectations:

- content is encrypted before storage
- backend should not have readable plaintext
- passwords/keys are required to decrypt content
- production deployment must use HTTPS

Out of scope for MVP:

- enterprise-grade audit controls
- malware scanning of encrypted files
- advanced abuse prevention

Nice to have later:

- rate limiting
- stronger content size controls
- CSP headers
- optional link key in URL fragment

That is the core product, and the MVP should stay focused on doing that well.
