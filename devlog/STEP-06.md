## Do This

### 1. Replace the placeholder in `app/models/`

This substep just prepares `app/models/` to become a real model package instead of an empty placeholder directory.

Right now, the backend has migration infrastructure but no actual application model. We are creating the files where the first persisted domain object will live.

From `backend/`:

```bash
rm app/models/.gitkeep
touch app/models/share.py
touch app/models/__init__.py
```

### 2. Create the `Share` model

This is the first real backend domain model.

What you are trying to achieve here:

- define one canonical database record for every share the app creates
- represent persisted backend data, not API request/response schemas
- capture the minimum fields the backend needs for text/file sharing, expiration, and burn-after-read

The `Share` object depends on:

- the shared SQLAlchemy `Base`
- SQLAlchemy column types
- `datetime` for timestamp fields

It should contain:

- `id`
- `type`
- `encrypted_payload`
- `file_name`
- `file_size`
- `mime_type`
- `expires_at`
- `burn_after_read`
- `is_deleted`
- `read_at`
- `created_at`

Write this code in `backend/app/models/share.py`:

```python
from datetime import datetime

from sqlalchemy import Boolean, DateTime, Integer, String, Text, func
from sqlalchemy.orm import Mapped, mapped_column

from app.db.base import Base


class Share(Base):
    __tablename__ = "shares"

    id: Mapped[str] = mapped_column(String(36), primary_key=True)
    type: Mapped[str] = mapped_column(String(20), nullable=False)
    encrypted_payload: Mapped[str] = mapped_column(Text, nullable=False)
    file_name: Mapped[str | None] = mapped_column(String(255), nullable=True)
    file_size: Mapped[int | None] = mapped_column(Integer, nullable=True)
    mime_type: Mapped[str | None] = mapped_column(String(255), nullable=True)
    expires_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), nullable=False)
    burn_after_read: Mapped[bool] = mapped_column(
        Boolean,
        nullable=False,
        default=False,
        server_default="false",
    )
    is_deleted: Mapped[bool] = mapped_column(
        Boolean,
        nullable=False,
        default=False,
        server_default="false",
    )
    read_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        nullable=False,
        server_default=func.now(),
    )
```

What this model represents:

- `id`: public share ID
- `type`: either `text` or `file`
- `encrypted_payload`: encrypted text body or encrypted file payload reference
- `file_name`, `file_size`, `mime_type`: file metadata when the share is a file
- `expires_at`: when the share becomes invalid
- `burn_after_read`: whether the share should only be used once
- `is_deleted`: soft-delete state
- `read_at`: when the share was first consumed
- `created_at`: creation timestamp

### 3. Register the model so Alembic can see it

Defining the model file is not enough by itself.

Alembic autogenerate works by inspecting `Base.metadata`, and `Base.metadata` only includes models that have actually been imported. So this substep makes sure Alembic loads the `Share` model before it reads metadata.

Write this code in `backend/app/models/__init__.py`:

```python
from app.models.share import Share

__all__ = ["Share"]
```

Then update `backend/alembic/env.py` so it imports the models package before `target_metadata` is read.

Add this import:

```python
from app import models  # noqa: F401
```

Place it with the other app imports, before:

```python
target_metadata = Base.metadata
```

Without this import, Alembic autogenerate may miss your model entirely.

### 4. Verify the model files

Before generating a migration, first confirm that the model and its registration are in place.

The goal here is to catch file/layout mistakes early instead of debugging Alembic output later.

From `backend/`:

```bash
find app/models -maxdepth 2 -type f | sort
sed -n '1,240p' app/models/share.py
sed -n '1,80p' app/models/__init__.py
sed -n '1,100p' alembic/env.py
```

Expected model file list:

```text
app/models/__init__.py
app/models/share.py
```

### 5. Generate the first real migration

Now that the `Share` model exists and Alembic can discover it, you want Alembic to compare:

- the current database state
- the current SQLAlchemy metadata

and generate the SQL changes needed to create the `shares` table.

From `backend/`:

```bash
uv run alembic revision --autogenerate -m "create shares table"
```

Expected output shape:

```text
INFO  [alembic.runtime.migration] Context impl PostgresqlImpl.
INFO  [alembic.runtime.migration] Will assume transactional DDL.
INFO  [alembic.autogenerate.compare] Detected added table 'shares'
Generating .../alembic/versions/<revision_id>_create_shares_table.py ...  done
```

Then list the generated file:

```bash
find alembic/versions -maxdepth 1 -type f | sort
```

You should now have:

- the generated revision file
- the old `.gitkeep`

At this point, remove `.gitkeep` because `alembic/versions/` is no longer empty:

```bash
rm alembic/versions/.gitkeep
```

### 6. Inspect the generated migration

Autogenerate is helpful, but you should still read the migration before applying it.

The point of this substep is to confirm that Alembic understood your model correctly and is about to create the table you expect, with the columns you expect.

Open the generated migration file:

```bash
sed -n '1,260p' alembic/versions/*_create_shares_table.py
```

Confirm it contains:

- `op.create_table("shares", ...)`
- all expected columns from the model
- `sa.PrimaryKeyConstraint("id")`

Do not rewrite it by hand unless something is clearly wrong.

The exact revision ID will vary, so the filename will be different on your machine.

### 7. Apply the migration

Once the migration file looks correct, apply it to the local database.

This is the moment where the model definition becomes a real database table in Postgres.

From `backend/`:

```bash
uv run alembic upgrade head
```

Expected output shape:

```text
INFO  [alembic.runtime.migration] Context impl PostgresqlImpl.
INFO  [alembic.runtime.migration] Will assume transactional DDL.
INFO  [alembic.runtime.migration] Running upgrade -> <revision_id>, create shares table
```

### 8. Verify the table exists

After the upgrade, do not just assume it worked.

This substep confirms two things:

- SQLAlchemy can see the `shares` table in the database
- Alembic records the new revision as the current head

From `backend/`:

```bash
uv run python -c "from sqlalchemy import inspect; from app.db.session import engine; print(inspect(engine).get_table_names())"
```

Expected output:

```text
['alembic_version', 'shares']
```

The order may vary slightly, but `shares` must be present.

Then verify Alembic state:

```bash
uv run alembic current
```

Expected output shape:

```text
INFO  [alembic.runtime.migration] Context impl PostgresqlImpl.
INFO  [alembic.runtime.migration] Will assume transactional DDL.
<revision_id> (head)
```
